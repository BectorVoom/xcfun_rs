# 00 — Overview

## 1. Purpose

`xcfun_rs` is a Rust-from-scratch reimplementation of the xcfun exchange–correlation (XC) functional library for density functional theory (DFT). It reproduces the full public C API of the C++ reference in `xcfun-master/` and extends it with native Rust bindings, a unified CPU/GPU evaluation backend implemented in `cubecl`, and build-time-guaranteed numerical parity with the reference library.

The reference deliverable of a correct implementation is: for every functional in the C++ library, for every `(xcfun_vars, xcfun_mode, order)` tuple the C++ library supports, and for every element of the output array, the Rust output agrees with the C++ output to relative error ≤ 1.0 × 10⁻¹².

## 2. Scope

### 2.1 In scope

| Item | Details |
|------|---------|
| Full public C API parity | Every symbol declared in `xcfun-master/api/xcfun.h` (see §3 below for the complete list) |
| All 78 functionals | `XC_SLATERX` through `XC_PW91C` from `functionals/list_of_functionals.hpp` |
| All named aliases | 50+ aliases (`lda`, `pbe`, `b3lyp`, `pbe0`, `camb3lyp`, `m06`, `scan`, `b97`, `ldaerf`, …) from `functionals/aliases.cpp` |
| All 4 tunable parameters | `EXX`, `RANGESEP_MU`, `CAM_ALPHA`, `CAM_BETA` |
| All 31 `xcfun_vars` combinations | LDA, GGA, metaGGA, 2nd-order Taylor; alpha or alpha/beta or n/s; with/without laplacian, kinetic, current, explicit gradient components |
| All 3 evaluation modes | `XC_PARTIAL_DERIVATIVES`, `XC_POTENTIAL`, `XC_CONTRACTED` |
| Derivative orders | 0…`XCFUN_MAX_ORDER` (6), with `XC_PARTIAL_DERIVATIVES` capped at 4 as in the reference |
| Native Rust API | Typed, owned handle, `Result<_, XcError>`, no hidden globals |
| C ABI | Drop-in replacement for `xcfun.h`, generated by `cbindgen` |
| Python bindings | Via `pyo3` 0.28 + `rust-numpy` 0.28; zero-copy for `f64` grids |
| CPU/GPU batch evaluation | Unified `cubecl` kernels usable with `CpuRuntime`, `CudaRuntime`, and `WgpuRuntime` (f64-capable devices only for Wgpu) |
| Validation harness | Compares the Rust implementation against the C++ reference across the full API, full order range, and representative density inputs |

### 2.2 Out of scope

| Item | Rationale |
|------|-----------|
| Changing the public API semantics | The API is a strict compatibility surface |
| Integrating a third-party AD framework (Enzyme, hyperdual, `ad`) | Reference accuracy requires algorithmic parity with xcfun's bit-flag multilinear polynomial representation; see 07-accuracy-strategy.md |
| Distributed (multi-node) evaluation | Density grids are local by construction; batch parallelism over a grid is already exposed through the cubecl batch API |
| Autodiff of user-supplied Rust functionals | Only the library's built-in functional set is differentiated |
| Symbolic output | Numerical `f64` only; rational / symbolic forms are not a user need for DFT grids |
| Differentiation order > 6 | The reference fixes `XCFUN_MAX_ORDER = 6`; raising it would require re-verifying the coefficient algebra |

## 4. Goals

| G‑id | Goal | Measurement |
|------|------|-------------|
| G1 | Numerical parity with C++ xcfun | For every built-in test vector in `functional_data.test_in / test_out`, and for a generated 10k-point regression grid, max relative error ≤ 1e-12 |
| G2 | C ABI drop-in | `cargo build -p xcfun-capi` produces a `.so`/`.dll`/`.dylib` whose `xcfun.h` symbol set and signatures match `xcfun-master/api/xcfun.h` byte-for-byte where feasible |
| G3 | Single kernel source for CPU and GPU | Every per-point evaluation routine is written once, as a `#[cube]` function, and dispatched to `CpuRuntime` or `CudaRuntime`/`WgpuRuntime` at runtime |
| G4 | Zero allocations on the per-point hot path | `xcfun_rs::Functional::eval` allocates no heap after construction (`#[no_std]`-compatible core crate) |
| G5 | Branch-light, cache-dense inner loop | `CTaylor<T, N>` stored as a single `[T; 1 << N]` with no indirection; functionals parameterised over `N` via const generics |
| G6 | Idiomatic Rust error model | `thiserror` for library errors, no panics outside contract violations, `Result<_, XcError>` returned from every fallible routine |
| G7 | Python parity | `xcfun_rs` Python module loads in CPython ≥ 3.10 and yields the same arrays as the C++ Python wrapper |

## 5. Non‑Goals

1. Bit-identical output with C++ xcfun. The goal is relative error ≤ 1e-12, not bit identity; floating-point reassociation across compilers and libraries makes bit identity not enforceable cross-platform.
2. Matching C++ xcfun internals line-for-line. The AD algebra must be algorithmically identical (see §7.3 of 07-accuracy-strategy.md); the rest of the library may be redesigned freely.
3. Supporting pre-`XCFUN_API_VERSION = 2` C headers. We target exactly the current API.
4. Extended precision (`f128`, `long double`). xcfun reference runs on `double`; `f64` is sufficient for the stated tolerance.
5. GPU-resident stateful functionals (persistent weights on device across multiple `eval_vec` calls). Weights are small (≤ `XC_NR_PARAMETERS_AND_FUNCTIONALS = 82` f64 values) and can be uploaded per launch without measurable cost.

## 6. Success Criteria

The project is considered complete when all of the following hold:

| Criterion | Verification |
|-----------|--------------|
| S1 | `cargo test --workspace --all-features` passes on Linux/macOS/Windows |
| S2 | The validation harness (see [09-testing-strategy.md](09-testing-strategy.md)) compares every `(functional, vars, mode, order)` tuple exercised by the C++ `xcfun_test` against the Rust implementation with `max |out_rust − out_cpp| / |out_cpp| ≤ 1e-12`. 0 failures required for merge to `main`. |
| S3 | Every alias in `xcfun-master/src/functionals/aliases.cpp` is resolvable and composes the same functional-weight set as the C++ implementation |
| S4 | `xcfun_rs::eval_vec` on a 100k-point GGA grid is within ±10 % of C++ wall-clock on the same machine, backend CPU |
| S5 | With `CudaRuntime`, the GPU evaluation of a 1M-point grid returns results within 1e-12 relative tolerance of the CPU evaluation of the same grid |
| S6 | C ABI golden tests: a C test program compiled against the Rust-generated `xcfun.h` and linked to the Rust `.so` produces the same outputs as the same program linked to the C++ library |
| S7 | No library crate depends on `anyhow`. `cargo metadata` | `jq`-filter verifies |
| S8 | `cargo deny check` passes for licenses and advisories |
| S9 | `cargo doc --no-deps` builds cleanly, with every public item documented |

## 7. Constraints Inherited from CLAUDE.md

- Rust Edition 2024 (MSRV 1.85+).
- `thiserror` 2.0.18 for library errors; `anyhow` 1.0 only at application boundaries (benches, examples, harness).
- `bitflags` 2.10.0 for `Dependency`.
- `cubecl` pinned at `=0.10.0-pre.3` with `cubecl-cuda` and `cubecl-wgpu` feature-gated.
- `cbindgen` 0.29.2 for header generation; `pyo3` 0.28.3 + `rust-numpy` 0.28.0 for Python.
- `criterion` 0.8.2 for benches; `approx` 0.5 for float comparison.

All architectural decisions in the rest of this document set respect these constraints.

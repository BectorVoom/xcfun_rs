# 05 â€” Module responsibilities

Each crate has exactly one responsibility, stated as a single sentence at the top of its entry. The rest of the entry lists the public surface (items re-exported or listed in `pub use`), the internal surface (reachable only within the crate), the seams for testing, and the inbound/outbound dependencies.

"Public" means visible outside the crate. "Internal" means `pub(crate)` or private. "Test seam" means an item exposed (possibly `#[cfg(test)]` or behind the `testing` feature) solely so that downstream tests can exercise the module in isolation.

## 1. `xcfun-ad`

**Responsibility**: provide the multilinear bit-flag-indexed Taylor polynomial `CTaylor<T, N>` and the scalar series expansions required to compose elementary functions on it.

| Surface | Items |
|---------|-------|
| Public | `CTaylor<T, N>`, `Num`, `CNST`, `VAR0..VAR6`, `Assert`, `expand::{inv, exp, log, pow, sqrt, erf}` |
| Internal | `ctaylor_mul` recursion helpers; scratch buffer types; const-generic dispatch shims |
| Test seam | `pub mod for_tests { pub use crate::expand::*; pub fn raw_coeffs<T, const N: usize>(t: &CTaylor<T, N>) -> &[T; 1 << N]; }` under `feature = "testing"` |
| Depends on | `core` only (no `std` when the `std` feature is off) |
| Allocates on hot path? | Never |
| Unsafe? | None; bit-index arithmetic is checked in debug, dropped in release |

Key invariant: `CTaylor::set(idx, v)` and `CTaylor::get(idx)` are both constant time and contain no branches beyond an array bounds check that the optimiser elides in release builds.

Property-test candidates:
- Ring axioms: commutativity/associativity/distributivity of `+` and `*` on `CTaylor<f64, N>` for `N âˆˆ 0..=6`.
- `exp(x) * exp(-x) â‰ˆ 1` within `1e-14` relative error for polynomials with random small coefficients.
- `log(exp(x)) â‰ˆ x`, `sqrt(x)^2 â‰ˆ x`, `pow(x, a) * pow(x, -a) â‰ˆ 1`.
- Derivative-of-product rule: `d/dxâ‚€ (fÂ·g) = f'Â·g + fÂ·g'` â€” verified by inspecting the bit-flag-0 coefficient.

## 2. `xcfun-core`

> **Updated 2026-04-22 (Phase 2 D-04).** Pre-Phase-2 this section described `xcfun-core` as the owner of functional bodies, `DensVars<T>`, the `Functional` struct, `eval` entry point, and the per-order scalar dispatcher. ## 2.1 `xcfun-eval`

> **Added 2026-04-22 (Phase 2 D-04).** This crate did not exist in the pre-Phase-2 design. It was introduced in Plan 02-03 to host the cubecl launcher + functional bodies + dispatch, cleanly separated from the cubecl-free `xcfun-core` types-and-registry crate.

**Responsibility**: the cubecl-bearing evaluation layer. Hosts per-functional `#[cube] fn` bodies, the `DensVarsDev<F>` cubecl type, the `build_densvars` + `regularize` kernels, the `dispatch_kernel` `#[cube]` comptime-if chain, and the `Functional` struct + `eval` entry point that the tier-1 self-tests and the tier-2 validation harness consume.

| Surface | Items |
|---------|-------|

| Depends on | `xcfun-core`, `xcfun-ad`, `cubecl = "=0.10.0-pre.3"`, `cubecl-cpu = "=0.10.0-pre.3"`, `thiserror` 2 |

## 3. `xcfun-kernels`

**Responsibility**: expose each per-point functional evaluation as a `#[cube]` function usable by `cubecl::CpuRuntime`, `CudaRuntime`, and `WgpuRuntime`.

| Surface | Items |
|---------|-------|
| Public | `eval_batch_kernel<R: cubecl::Runtime, const VARS: u32, const ORDER: u32>` (the single `#[cube(launch_unchecked)]` entry point), plus per-functional `#[cube]` helpers declared inside the crate |
| Internal | Dispatch table mapping `(FunctionalId, order)` â†’ `#[cube] fn` pointer (implemented as a `match` inside the kernel to keep the host/device boundary closed); scratch buffer sizing helpers |
| Test seam | `pub mod for_tests { pub fn launch_on_cpu_runtime(...) -> Vec<f64>; }` â€” under `feature = "cpu-testing"`, launches the kernel with `CpuRuntime` and returns the host-side result |
| Depends on | `xcfun-core` (for `Vars`, layout tables, `DensVars` builder used on the scalar path), `cubecl` |
| Allocates on hot path? | Device buffers only; no host heap |
| Unsafe? | `launch_unchecked` uses `unsafe` at the entry point as per cubecl idiom; bounds-checked inside |

Purity rule: inside a `#[cube]` body, only other `#[cube]`-annotated functions and `cubecl`-provided math intrinsics may be called. Each functional is mirrored by a `#[cube]`-annotated version using the generic `F: Float` (cubecl) through a `Num` implementation for `F`; see [06-cubecl-strategy.md](06-cubecl-strategy.md).

Property-test candidates:
- Parity: for a fixed seed RNG density grid, `eval_batch_kernel<CpuRuntime, V, K>` result matches `xcfun-core::Functional::eval` to within `5e-15` relative error (sanity bound, not the headline 1e-12 parity against C++).

## 4. `xcfun-gpu`

**Responsibility**: own the `Batch` lifecycle â€” backend selection, device buffer pooling, hostâ†”device transfers, and asynchronous launch.

| Surface | Items |
|---------|-------|
| Public | `Backend` (enum `Cpu`, `Cuda`, `Wgpu`), `Batch<'fun, R>`, `auto_backend()`, `cpu_client()`, `cuda_client(device_id)`, `wgpu_client(device)` |

| Allocates on hot path? | Only device buffers; host heap used during `reserve()` |
| Unsafe? | `launch_unchecked` calls and a single `Send`-bound escape hatch in the buffer pool, both documented |

Decision tree for `auto_backend()`:

```
if env XCFUN_FORCE_BACKEND is set â†’ use it (panic if unsupported)
else if CudaRuntime::is_available() â†’ Cuda
else if WgpuRuntime::is_available() && wgpu_supports_f64(device) â†’ Wgpu
else â†’ Cpu
```

Property-test candidates: backend-agnostic equivalence â€” fixed input batch, identical output across `Cpu`, `Cuda`, `Wgpu` (f64-capable) within 1e-12.

## 6. `xcfun-capi`

**Responsibility**: implement the C ABI declared in `xcfun-master/api/xcfun.h`, generate a matching `xcfun.h` via `cbindgen`, and package as `cdylib` + `staticlib`.

| Surface | Items |
|---------|-------|
| Public (to C) | Every declaration in `xcfun.h`; no Rust item is `pub` outside the `extern "C"` block |
| Public (to Rust) | Only the crate's build script helpers (for the `cbindgen` step) |
| Internal | `xcfun_t` wrapper; a `catch_unwind` shim around every entry point to convert Rust panics to a `xcfun::die`-equivalent `abort` |
| Test seam | A C file `tests/c_abi.c` compiled by `cc` in `build.rs`, linked against the staticlib, and run as a cargo test |
| Depends on | `xcfun-rs`, `cbindgen` (build-dep) |
| Allocates on hot path? | Never beyond `xcfun_new` / `xcfun_delete` |
| Unsafe? | Entire crate is `unsafe`-dense by necessity (`extern "C"` + raw pointers); every `unsafe` block documents its preconditions |

Panic policy: any panic inside a C entry point is caught and converted to `xcfun::die`-style behaviour (stderr message, `abort`). This matches the reference: the C++ library calls `std::abort()` through `xcfun::die` in pathological cases.

## 10. Module boundary rules

> **Updated 2026-04-22 (Phase 2 D-04).** Rules 2, 3, and 10 below reflect the cubecl-pivot boundary reshuffle. `xcfun-core` is now narrowed to types + registry tables (no evaluation); `xcfun-eval` hosts the cubecl launcher, functional bodies, and `Functional` struct; `xcfun-kernels` (if a future phase introduces it) and `xcfun-gpu` build on top of `xcfun-eval`. Enforced by `cargo xtask check-boundaries` (QG-02).

1. `xcfun-ad` is a pure Phase-1 cubecl-bearing math crate owning `CTaylor<F, N>` + the `ctaylor_*` + `*_expand` primitives. It must never know about `Vars`, `Functional`, `DensVarsDev`, or `FunctionalDescriptor`.
2. `xcfun-core` owns the static type surface + registry tables (FUNCTIONAL_DESCRIPTORS, VARS_TABLE, ALIASES). Cubecl-free. Does not depend on `cubecl` or `xcfun-ad` (verified by `xtask check-boundaries` + Plan 02-01 Wave-0e manifest diff). No other crate defines a `FunctionalDescriptor`.
3. `xcfun-eval` is the cubecl launcher. Hosts per-functional `#[cube] fn <name>_kernel` bodies, `DensVarsDev<F>`, `build_densvars`, `regularize`, the `dispatch_kernel` comptime if-chain, and the `Functional` struct + `eval` entry point (D-21). Depends on `xcfun-core` (types + registry) + `xcfun-ad` (cubecl primitives) + `cubecl` + `cubecl-cpu`. Never depends on `xcfun-gpu`.
4. `xcfun-gpu` (future Phase 6) owns CUDA / Wgpu runtimes + the `Batch` lifecycle + device-buffer pools. Consumes `xcfun-eval` + `xcfun-core`. Adds `cubecl-cuda` / `cubecl-wgpu` feature-gated.
5. `xcfun-rs` (future Phase 5) is a faÃ§ade. Re-exports from `xcfun-eval` + `xcfun-gpu`. No logic lives there.
6. `xcfun-capi` (future Phase 5) is a C ABI shim. No logic lives there.
7. `xcfun-py` (future Phase 7) is a Python shim. No logic lives there.
8. The C++ reference lives only in `validation/` (tier-2 parity harness; landed in Plan 02-06) and in `xcfun-master/` (vendored source). No production crate calls into the C++ library.
9. `anyhow` is permitted **only** in `validation/`, `xtask/`, benches, examples, and dev-deps (QG-01). The library graph (`xcfun-ad`, `xcfun-core`, `xcfun-eval`, and future `xcfun-gpu`/`xcfun-rs`/`xcfun-capi`/`xcfun-py`) uses `thiserror` for error types only.
10. Functional surface ownership (post D-04): per-functional `#[cube] fn <name>_kernel` bodies + the `Functional` struct + dispatch all live in `xcfun-eval`. `xcfun-core` carries no evaluation code. This is enforced structurally by (a) `xcfun-core` not depending on `cubecl`, (b) the `FUNCTIONAL_DESCRIPTORS` table storing only metadata (no function pointers), and (c) `xtask check-boundaries` asserting each crate's `[dependencies]` against an allowlist.

These boundaries are enforced by the `cargo xtask check-boundaries` step in CI: the script parses each crate's `Cargo.toml` and asserts the dependency set against an allowlist keyed on the crate name.

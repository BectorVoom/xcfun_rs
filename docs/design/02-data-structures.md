# 02 — Data structures

Every data structure on the evaluation hot path is stack-allocated and `#[repr(C)]` or `#[repr(transparent)]` where a C ABI is required. The only heap data lives at construction time (functional registry) or at batch boundaries (device buffers).

Conventions:
- `Num` is a trait defined in `xcfun-ad` (see §1.2).
- `CTaylor<T, const N: usize>` is the AD polynomial; `T: Num`, `N` is the number of tensored variables (bit positions), with `N ≤ 7`.
- No data structure allocates on any call path downstream of `Functional::eval`.

---

## 1. `xcfun-ad` — automatic differentiation primitives

### 1.1 `CTaylor<T, const N: usize>`

Bit-flag-indexed multilinear polynomial. Each of `N` bits in the index corresponds to "include a first-derivative factor with respect to variable i". The coefficient at index `I` with popcount `k` is the k-th mixed partial derivative with respect to the variables flagged in `I`. This matches xcfun's `ctaylor` (`xcfun-master/src/taylor/ctaylor.hpp`, lines 154-357).

```rust
// crate: xcfun-ad
#[repr(C)]
#[derive(Copy, Clone, Debug)]
pub struct CTaylor<T: Num + Copy, const N: usize>
where
    // compile-time constraint: need 1 << N coefficients.
    // N is bounded to 0..=7 by the const assertion below.
    Assert<{ N <= 7 }>: True,
{
    pub c: [T; 1 << N],
}
```

Layout:

| Property | Value |
|----------|-------|
| Size | `(1 << N) * size_of::<T>()` |
| Alignment | `align_of::<T>()` |
| For `T = f64`, `N = 6` | 64 × 8 = 512 bytes (fits in 2 cache lines on a typical GPU warp) |
| Allocation | Stack always; never boxed on the hot path |
| Heap? | Never |

The `Assert<{ N <= 7 }>: True` is a const-generics trick to fail compilation for `N > 7`. In the reference, `XCFUN_MAX_ORDER = 6`, plus one additional bit for `XC_CONTRACTED` handling → 7 bits max.

**Index semantics** (identical to C++ `VAR0..VAR7`, `CNST`):

| Const | Value | Meaning |
|-------|-------|---------|
| `pub const CNST: usize = 0;` | `0` | Constant term (function value) |
| `pub const VAR0: usize = 1;` | `0b0000_0001` | Derivative w.r.t. first active variable |
| `pub const VAR1: usize = 2;` | `0b0000_0010` | Derivative w.r.t. second active variable |
| `pub const VAR2: usize = 4;` | `0b0000_0100` | Derivative w.r.t. third active variable |
| … | | |
| `pub const VAR6: usize = 64;` | `0b0100_0000` | Derivative w.r.t. seventh active variable |

Extracting the mixed partial derivative ∂ᵏf/∂x_{i₁}…∂x_{iₖ} is a direct array index: `t.c[VAR_i1 | VAR_i2 | ... | VAR_ik]`. No arithmetic, no branching. This is the critical design property that makes per-point evaluation branch-light.

**Public constructors** (all stack-allocated, all `#[inline]`):

```rust
impl<T: Num + Copy, const N: usize> CTaylor<T, N>
where Assert<{ N <= 7 }>: True
{
    /// Zero polynomial.
    pub fn zero() -> Self { Self { c: [T::ZERO; 1 << N] } }

    /// Constant polynomial: f(x) = value, all derivatives zero.
    pub fn from_const(value: T) -> Self {
        let mut c = [T::ZERO; 1 << N];
        c[CNST] = value;
        Self { c }
    }

    /// Construct `value + var_slot * slope`, used when seeding an input with
    /// a first-order derivative. Mirrors the three-argument ctaylor(x, k, dx)
    /// constructor in ctaylor.hpp.
    pub fn seed(value: T, var_slot: usize, slope: T) -> Self {
        debug_assert!(var_slot < (1 << N));
        debug_assert!(var_slot.count_ones() == 1);
        let mut c = [T::ZERO; 1 << N];
        c[CNST] = value;
        c[var_slot] = slope;
        Self { c }
    }

    /// Set a single coefficient (used by the dispatcher when seeding inputs).
    pub fn set(&mut self, idx: usize, value: T) { self.c[idx] = value; }

    /// Read a coefficient (derivative extraction).
    pub fn get(&self, idx: usize) -> T { self.c[idx] }
}
```

**Algebra** (all `#[inline]`, all in `ctaylor.rs` / `ctaylor_mul.rs`):

- `Add`, `Sub`, `Neg` — elementwise, `(1 << N)` fused additions.
- `Mul` — multilinear convolution over bit-flag indices. Unrolled per `N` at compile time (implemented via recursion on `N` à la `ctaylor_rec<T,Nvar>::multo` in the reference, lines 43-59, 118-142 of `ctaylor.hpp`).
- `Mul<T>` / `Mul<f64>` — scalar multiplication, `(1 << N)` scalar multiplies.
- `Div`, `reciprocal` — Taylor series composition with `inv_expand`.
- `exp`, `log`, `pow`, `sqrt`, `erf` — series composition with the corresponding `*_expand` table.

The `Mul` operation is written once, parameterised over `N`, using a const-generic recursion:

```rust
#[inline]
pub fn mul_into<const N: usize>(dst: &mut CTaylor<f64, N>,
                                a: &CTaylor<f64, N>,
                                b: &CTaylor<f64, N>)
where Assert<{ N <= 7 }>: True
{
    // P_N = P_{N-1} + x_N * R_{N-1}  (matches ctaylor_rec::multo)
    // Generated by a const-generic helper; body unrolls for N ∈ {0..=6}.
}
```

Internal scratch arrays (series coefficients) are also stack-allocated `[T; N+1]`.

### 1.2 `Num` trait

```rust
// crate: xcfun-ad
pub trait Num:
    Copy
    + core::ops::Add<Output = Self>
    + core::ops::Sub<Output = Self>
    + core::ops::Mul<Output = Self>
    + core::ops::Div<Output = Self>
    + core::ops::Neg<Output = Self>
{
    const ZERO: Self;
    const ONE: Self;

    fn reciprocal(self) -> Self;
    fn sqrt(self) -> Self;
    fn exp(self) -> Self;
    fn log(self) -> Self;
    fn pow(self, exponent: Self) -> Self;
    fn powi(self, exponent: i32) -> Self;
    fn erf(self) -> Self;
    fn asinh(self) -> Self;
    fn atan(self) -> Self;
}
```

`Num` is implemented for `f64` (CPU scalar path), for `CTaylor<f64, N>` (derivative paths), and for `cubecl::prelude::f64` (device path). Every functional is written generic over `T: Num` so the same source compiles for both scalar evaluation and automatic differentiation.

### 1.3 Expansion tables (`xcfun-ad::expand::*`)

Each `*_expand<T, NDEG>(out: &mut [T; NDEG + 1], x0: T)` writes the scalar Taylor-series coefficients used by series composition. They are the bit-for-bit analogues of `tmath.hpp`:

- `inv_expand::fill::<NDEG>(out, a)` — coefficients for `1 / (a + x)`.
- `exp_expand::fill::<NDEG>(out, x0)` — `exp(x0 + x)`.
- `log_expand::fill::<NDEG>(out, x0)` — `log(x0 + x)`.
- `pow_expand::fill::<NDEG>(out, x0, a)` — `(x0 + x)^a`.
- `sqrt_expand::fill::<NDEG>(out, x0)` — special case of `pow`.
- `erf_expand::fill::<NDEG>(out, a)` — via `gauss_expand` + term-by-term integration.

All four are `const fn` compatible where feasible, and all use `f64` libm internally on CPU or the device-provided intrinsics when invoked from a `#[cube]` context.

| Object | Size (bytes) | Storage |
|--------|--------------|---------|
| `CTaylor<f64, 0>` | 8 | stack |
| `CTaylor<f64, 1>` | 16 | stack |
| `CTaylor<f64, 2>` | 32 | stack |
| `CTaylor<f64, 3>` | 64 | stack |
| `CTaylor<f64, 4>` | 128 | stack |
| `CTaylor<f64, 5>` | 256 | stack |
| `CTaylor<f64, 6>` | 512 | stack |

## 2. `xcfun-core::depends::Dependency`

```rust
// crate: xcfun-core
use bitflags::bitflags;

bitflags! {
    /// Bit set of the raw density ingredients a functional needs.
    ///
    /// Mirrors `XC_DENSITY | XC_GRADIENT | XC_LAPLACIAN | XC_KINETIC | XC_JP`
    /// in `xcfun-master/src/xcint.hpp` lines 44-48.
    #[repr(transparent)]
    #[derive(Copy, Clone, PartialEq, Eq, Debug, Default)]
    pub struct Dependency: u8 {
        const DENSITY   = 1 << 0;  //  1 — rho, rho_alpha, rho_beta
        const GRADIENT  = 1 << 1;  //  2 — gamma / gradient components
        const LAPLACIAN = 1 << 2;  //  4 — nabla^2 rho
        const KINETIC   = 1 << 3;  //  8 — tau
        const JP        = 1 << 4;  // 16 — current density (JPAA, JPBB)
    }
}
```

`#[repr(transparent)]` over `u8` guarantees single-byte representation, bit-identical with the C header values. Layout-critical: composed into `FunctionalDescriptor.depends` and `Functional.depends` and compared with `VarsTable[vars].provides`.

## 5. `xcfun-eval::density_vars::DensVarsDev<F>`

> **Updated 2026-04-22 (Phase 2 D-02 + D-04).** Pre-Phase-2 this section described a host `DensVars<T: Num>` struct of 29 `T`-typed fields living in `xcfun-core::densvars`. Phase 2 Plan 02-03 (Wave-1B-1) landed a cubecl-native redesign in a different crate. The old host-struct design is **SUPERSEDED**; the as-built shape is `DensVarsDev<F>` as a `#[derive(CubeType, CubeLaunch)]` type in `crates/xcfun-eval/src/density_vars.rs`. The reason for the move is D-04 (functional bodies + density-var construction must be `#[cube]` so a single source compiles to CPU + GPU) and D-02 (cubecl nesting spike confirmed Pattern A — multi-`Array<F>` nested cube types are supported on cubecl 0.10-pre.3). See the Phase 2 CONTEXT doc §Decisions D-01..D-04 and Plan 02-03 SUMMARY for rationale.

### 5.1 `DensVarsDev<F>` — cubecl-native density-variables aggregate

```rust
// crate: xcfun-eval (NOT xcfun-core — per D-04, xcfun-core is cubecl-free)
// module: crates/xcfun-eval/src/density_vars.rs
use cubecl::prelude::*;

#[derive(CubeType, CubeLaunch)]
pub struct DensVarsDev<F: Float> {
    // Raw inputs per the 31 Vars variants (pre-seeded CTaylor coefficients — see §5.3)
    pub a: Array<F>,  pub b: Array<F>,
    pub gaa: Array<F>, pub gab: Array<F>, pub gbb: Array<F>,
    pub lapa: Array<F>, pub lapb: Array<F>,
    pub taua: Array<F>, pub taub: Array<F>,
    pub jpaa: Array<F>, pub jpbb: Array<F>,

    // Derived quantities (eager; populated by build_densvars inside the kernel)
    pub n: Array<F>,     // a + b
    pub s: Array<F>,     // a - b
    pub gnn: Array<F>,   // gaa + 2*gab + gbb
    pub gss: Array<F>,   // gaa - 2*gab + gbb
    pub gns: Array<F>,   // gaa - gbb
    pub lapn: Array<F>, pub laps: Array<F>,
    pub tau: Array<F>, pub taun: Array<F>, pub taus: Array<F>,
    pub zeta: Array<F>,  // s/n
}
```

The Phase-2 slice is **22 named `Array<F>` fields** (Plan 02-03 landed this exact set). The set may extend in Phase 3 (`r_s`, `n_m13`, `a_43`, `b_43`, and the gradient-component scalars `ax, ay, az, bx, by, bz, ...`) as GGA functionals need them; the `#[derive(CubeType, CubeLaunch)]` layer accepts new fields without breaking the launch ABI because cubecl regenerates the launcher.

Each `Array<F>` holds the full `[F; 1 << N]` CTaylor coefficient vector for that density variable, backed by the cubecl launcher. Inside a `#[cube] fn` body the field is read through `d.a` and passed to Phase-1 `ctaylor_*` primitives directly — no host materialisation.

Source of truth: `crates/xcfun-eval/src/density_vars.rs` (field set), `crates/xcfun-eval/src/density_vars/build.rs` (builder chain), `crates/xcfun-eval/src/density_vars/regularize.rs` (regularize `#[cube] fn`).

### 5.2 Construction — `#[cube] fn build_densvars`

```rust
// crate: xcfun-eval
// module: crates/xcfun-eval/src/density_vars/build.rs

#[cube]
pub fn build_densvars<F: Float>(
    input: &Array<F>,                // inlen × (1 << N) flat array of pre-seeded CTaylor coefficients
    d: &mut DensVarsDev<F>,
    #[comptime] vars: u32,           // Vars discriminant
    #[comptime] n: u32,              // CTaylor order exponent
) {
    // comptime if-chain dispatcher — Phase 2 ships two variant arms:
    //   * XC_A_B (id = Vars::AB discriminant)
    //   * XC_A_B_GAA_GAB_GBB (id = Vars::ABGaaGabGbb discriminant)
    // Phase 3 extends with GGA-gradient arms; Phase 4 with metaGGA laplacian/kinetic arms.
    if vars == VARS_XC_A_B {
        build_xc_a_b::<F>(input, d, n);
    } else if vars == VARS_XC_A_B_GAA_GAB_GBB {
        build_xc_a_b_gaa_gab_gbb::<F>(input, d, n);
    }
    // NOTE: explicit helper-function chain replaces C-style fallthrough
    // (CORE-05 + Pitfall P5). Each arm writes every relevant field of `d`.

    regularize::<F>(d, n);           // CORE-06 / D-22 — see §5.3
}
```

### 5.3 Regularization — `#[cube] fn regularize`

```rust
// crate: xcfun-eval
// module: crates/xcfun-eval/src/density_vars/regularize.rs

const TINY_DENSITY: f64 = 1e-14;  // PhysicaL constant (f64; NEVER f32 — Plan 02-06 fix)

#[cube]
pub fn regularize<F: Float>(d: &mut DensVarsDev<F>, #[comptime] n: u32) {
    // Per CORE-06 + D-22: modify ONLY Array<F>[0] (the CNST coefficient).
    // All higher-order CTaylor coefficients (Array<F>[1..(1 << N)]) are preserved.
    // Clamps `a` and `b` to TINY_DENSITY where the constant term is below threshold.
    // Mirrors C++ `regularize(ctaylor<T,N> & x)` in densvars.hpp:22-30.
}
```

The `TINY_DENSITY` constant is declared in f64 (not f32) — Plan 02-06's Fix 1 established that even `F::new(1e-14_f32)` truncates to `9.99999982e-15` and cascades to 1.75e-8 rel-err, violating the 1e-12 contract. Every physical constant in `xcfun-eval` now uses `F::cast_from(f64_literal)` with an inline derivation comment.

The regularize invariant is guarded by a dedicated integration test (`crates/xcfun-eval/tests/regularize_invariant.rs`) exercising a CTaylor<f64, 2> input with a seeded first-derivative coefficient; the test asserts that coefficients at indices 1..=3 are unchanged pre/post regularize while index 0 is clamped.

### 5.4 Memory layout notes

Layout:

| Property | Value |
|----------|-------|
| Fields | 22 `Array<F>` fields (Phase 2); extends in Phase 3/4 |
| For `F = f64`, `N = 2` (order-2 PartialDerivatives) | 22 × 4 × 8 = 704 B of pre-seeded coefficients per point |
| `#[repr]` | cubecl-derived — launcher layout managed by `#[derive(CubeLaunch)]` |
| Heap | Only at batch-open (cubecl `Array<F>` allocation); zero-heap on the per-point hot path |

The per-point stack-frame budget is determined by CTaylor scratch allocations inside each functional `#[cube] fn`, not by `DensVarsDev` itself — cubecl keeps `Array<F>` handles as thin descriptors.

## 11. Summary of heap-allocation rules

| Location | Heap allocation? | Justification |
|----------|------------------|---------------|
| `xcfun-ad` | Never | All polynomial types are `[T; 1<<N]` stack arrays |
| `xcfun-core::DensVars` | Never | Stack struct; construction is straight-line stores |
| `xcfun-core::Functional::eval` (scalar CPU path) | Never | Uses stack `CTaylor` and fixed-size loops over `active_ids` |
| `xcfun-gpu::Batch::new` | Once per `Batch` | Device buffer allocation is unavoidable; pooled inside `Batch` |
| `xcfun-gpu::Batch::eval_vec` | Never | Reuses pooled buffers |
| `xcfun-rs::Functional::new` | Once (the `Box<Functional>`) | Required for `xcfun_t *` opacity |
| `xcfun-capi` layer | Never beyond `new` / `delete` | Thin transmute to `Functional` |
| `xcfun-py` | GIL-governed, never on hot eval path | numpy arrays supplied by caller |

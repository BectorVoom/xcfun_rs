# 09 — Testing strategy

Four tiers, each with a different failure mode in mind. Tier 1 catches code-gen regressions. Tier 2 catches algorithmic divergence from C++. Tier 3 catches cubecl/backend drift. Tier 4 catches algebraic identity violations.

All tiers run on CI; pull-request gates require Tiers 1, 2, 4 to pass. Tier 3 (GPU backends) runs nightly.

---

## 1. Test runner

`cargo-nextest`. Chosen for parallelism, test timeouts, and deterministic output. Configuration lives in `.config/nextest.toml`:

```toml
[profile.default]
retries = 0
threads = "num-cpus"
slow-timeout = { period = "30s", terminate-after = 3 }
```

For Python: `pytest` run via `maturin develop && pytest crates/xcfun-py/tests/`.

## 2. Tier 1 — per-functional self-tests (unit tests)

Source of truth: each reference functional declaration in `xcfun-master/src/functionals/*.cpp` contains `test_in[]`, `test_out[]`, `test_threshold`, `test_vars`, `test_mode`, `test_order`. These are encoded into Rust as part of `FUNCTIONAL_DESCRIPTORS` (see [02-data-structures.md §6](02-data-structures.md)).

Test file: `crates/xcfun-core/tests/functional_self_tests.rs`.

```rust
#[rstest]
#[case(FunctionalId::XC_SLATERX)]
#[case(FunctionalId::XC_VWN5C)]
// ...
// generated by xtask; one case per functional
fn functional_self_test(#[case] id: FunctionalId) {
    let desc = &FUNCTIONAL_DESCRIPTORS[id as usize];
    let mut fun = Functional::new();
    fun.set(desc.name, 1.0).unwrap();
    fun.eval_setup(desc.test_vars, desc.test_mode, desc.test_order as u32).unwrap();

    let mut out = vec![0.0; fun.output_length()];
    fun.eval(&desc.test_in, &mut out).unwrap();

    for (i, (&got, &want)) in out.iter().zip(desc.test_out.iter()).enumerate() {
        let rel_err = (got - want).abs() / want.abs().max(1.0);
        assert!(rel_err <= desc.test_threshold,
                "functional {}: element {}: rel_err {} > threshold {}",
                desc.name, i, rel_err, desc.test_threshold);
    }
}
```

Every functional ships its own test vector. Running `cargo test -p xcfun-core` takes < 3 seconds and catches structural code-gen regressions.

## 3. Tier 2 — C++-parity harness

Binary crate: `validation/` (the only consumer of `xcfun-master` at runtime).

### 3.1 Architecture

```
validation/
├── build.rs              # compiles xcfun-master into a static library (cc + C++ sources)
├── src/
│   ├── main.rs           # CLI: --order N --backend cpu|cuda|wgpu --filter <glob>
│   ├── cpp_shim.rs       # FFI wrappers: xcfun_new, xcfun_set, xcfun_eval, xcfun_delete
│   ├── fixtures.rs       # Density grid generators (seeded RNG)
│   ├── compare.rs        # Rel-error reducer
│   └── report.rs         # HTML + JSONL reporter
├── fixtures/
│   └── grid_10k_seed_0x1234abcd.jsonl.gz
└── Cargo.toml            # [dependencies] anyhow, approx, cc, serde_json, tracing-subscriber
```

### 3.2 Fixtures

The grid generator produces density vectors that exercise:
- Small densities (ρ ∈ [1e-6, 1e-2]) — regularisation path.
- Moderate densities (ρ ∈ [0.01, 10]) — typical molecular region.
- Fully polarised / unpolarised limits (|s/n| ∈ {1−ε, 0}).
- High gradient (χ² = γ / n^(8/3) ∈ [1, 1000]) — GGA enhancement limits.
- Large kinetic energy density (τ ≫ τ_W) — metaGGA stress.

The seeded `rand_xoshiro::Xoshiro256StarStar` RNG produces bit-identical streams across platforms; the seed is stored in the fixture filename.

### 3.3 Run

```bash
cargo xtask validate --order 0 --filter '*'       # ~2 min
cargo xtask validate --order 4 --filter '*'       # ~15 min on CI
cargo xtask validate --order 2 --filter 'm06*'    # focused debug
```

For each functional × vars × mode × order × point, the harness computes:

```
rel_err[i] = |out_rust[i] - out_cpp[i]| / max(|out_cpp[i]|, 1.0)
```

and the test fails if `max(rel_err) > 1e-12`.

### 3.4 Report

HTML report (`validation/report.html`) groups failures by functional, then by element index. JSONL (`validation/report.jsonl`) carries machine-readable records for downstream ingestion.

CI artefact retention: 30 days on green runs, unlimited on red runs (so post-hoc investigation is possible).

## 4. Tier 3 — cross-backend parity

For the `Cuda` and `Wgpu` backends, the test harness replays the Tier 2 density grid through `xcfun-gpu::Batch` and compares against the CPU output:

```
max_rel_err(cpu_result, gpu_result) ≤ 1e-13
```

This is tighter than the C++-parity tolerance because the computation path inside `xcfun_rs` is identical; the only variance is transcendental-function ULP.

Conditional exclusion for Wgpu:
- Functionals using `erf` (range-separated) are excluded from the Wgpu backend; if the harness encounters one, it logs a skip.

GPU-specific runners on GitHub Actions: `self-hosted, cuda` and `self-hosted, wgpu-f64` labels. Nightly schedule.

## 5. Tier 4 — property tests

`proptest` in `xcfun-ad/tests/proptest_algebra.rs` and `xcfun-core/tests/proptest_functional.rs`.

### 5.1 AD ring axioms

```rust
proptest! {
    #[test]
    fn taylor_mul_commutative(a in ctaylor_strategy::<3>(),
                              b in ctaylor_strategy::<3>()) {
        let ab = a * b;
        let ba = b * a;
        for i in 0..(1 << 3) {
            prop_assert!((ab.c[i] - ba.c[i]).abs() <= 2.0 * f64::EPSILON * ab.c[i].abs().max(1.0));
        }
    }

    #[test]
    fn taylor_exp_log_roundtrip(x in ctaylor_positive_strategy::<4>()) {
        let r = x.log().exp();
        for i in 0..(1 << 4) {
            prop_assert!((r.c[i] - x.c[i]).abs() <= 1e-13 * x.c[i].abs().max(1.0));
        }
    }
}
```

### 5.2 Functional linearity

```rust
proptest! {
    #[test]
    fn weight_linearity(rng_input in density_strategy()) {
        let mut fun_a = Functional::new();
        fun_a.set("slaterx", 1.0).unwrap();
        fun_a.eval_setup(Vars::AB, Mode::PartialDerivatives, 0).unwrap();

        let mut fun_b = Functional::new();
        fun_b.set("vwn5c", 1.0).unwrap();
        fun_b.eval_setup(Vars::AB, Mode::PartialDerivatives, 0).unwrap();

        let mut fun_ab = Functional::new();
        fun_ab.set("slaterx", 0.8).unwrap();
        fun_ab.set("vwn5c", 0.2).unwrap();
        fun_ab.eval_setup(Vars::AB, Mode::PartialDerivatives, 0).unwrap();

        let out_a = eval_single(&fun_a, &rng_input);
        let out_b = eval_single(&fun_b, &rng_input);
        let out_ab = eval_single(&fun_ab, &rng_input);

        let expected = 0.8 * out_a + 0.2 * out_b;
        prop_assert!((out_ab - expected).abs() <= 10.0 * f64::EPSILON * expected.abs().max(1.0));
    }
}
```

## 6. C ABI parity

`crates/xcfun-capi/tests/c_abi.c` is a C program that:
1. Loads the staticlib.
2. Invokes every C entry point at least once.
3. Compares results against a reference recording (generated by `xtask` against the C++ library).

`crates/xcfun-capi/build.rs` compiles the test via `cc::Build`, and `tests/c_abi.rs` is a thin Rust harness that runs the compiled binary and asserts its exit status.

`crates/xcfun-capi/tests/headers_match.rs`:

```rust
#[test]
fn generated_header_matches_reference() {
    let generated = fs::read_to_string("include/xcfun.h").unwrap();
    let reference = fs::read_to_string("../../xcfun-master/api/xcfun.h").unwrap();
    let gen_normalised  = normalise_comments_and_whitespace(&generated);
    let ref_normalised  = normalise_comments_and_whitespace(&reference);
    assert_eq!(gen_normalised, ref_normalised, "cbindgen output drift detected");
}
```

The normalisation strips comments and whitespace but retains symbol order, so changes to function signatures are caught while gratuitous comment diffs are ignored.

## 7. Python parity

`crates/xcfun-py/tests/test_parity.py` uses `pytest` and `hypothesis`:

```python
import numpy as np
import pytest
import xcfun_rs

def test_blyp_scalar_matches_reference():
    fun = xcfun_rs.Functional()
    fun.set("blyp", 1.0)
    fun.eval_setup(xcfun_rs.Vars.ABGaaGabGbb, xcfun_rs.Mode.PartialDerivatives, 1)
    density = np.array([0.39, 0.38, 0.1, 0.05, 0.1])
    out = fun.eval(density)
    # Against a recorded reference from the C++ Python wrapper
    expected = np.load("fixtures/blyp_order1_abgaagabgbb.npy")
    np.testing.assert_allclose(out, expected, rtol=1e-12)
```

When the C++ Python wrapper is installed on the test environment (`pip install xcfun`), the tests also run side-by-side against it.

## 8. Benchmarks

`criterion` 0.8 benches in each crate under `crates/*/benches/`. Bench harnesses cover:

- `xcfun-ad::ctaylor_mul` for `N ∈ {2, 3, 4, 5, 6}`.
- `xcfun-core::eval` scalar path for a representative LDA, GGA, metaGGA functional.
- `xcfun-gpu::eval_vec` for `nr_points ∈ {100, 10_000, 1_000_000}` on `Cpu`, `Cuda`.

Baselines stored per commit under `target/criterion/*`; regression rule: > 10 % slowdown on the hot bench (`xcfun-core::eval` over a 10k-point LDA) blocks merge.

## 9. Testability hooks (per crate)

| Crate | Testing feature | What it exposes |
|-------|-----------------|-----------------|
| `xcfun-ad` | `feature = "testing"` | `for_tests::raw_coeffs`, synthesis constructors for property tests |
| `xcfun-core` | `feature = "testing"` | `for_tests::set_descriptor` (inject mocks), `for_tests::eval_partial_n` (bypass dispatch) |
| `xcfun-kernels` | `feature = "cpu-testing"` | `for_tests::launch_on_cpu_runtime` — launches the kernel on `CpuRuntime` with no GPU hardware needed |
| `xcfun-gpu` | `feature = "metrics"` | `BatchMetrics` with transfer/compute timings |
| `xcfun-capi` | `feature = "panic-hook"` | Installs a panic hook that writes to a known path (for test-driven panic detection) |
| `xcfun-py` | None — tested via Python |

These features are off by default in release builds; CI enables them as needed.

## 10. Static analysis

- `cargo clippy --workspace --all-features -- -D warnings` blocks merge on any lint.
- `cargo fmt --check` blocks merge on formatting drift.
- `cargo deny check` blocks merge on license / advisory violations.
- `cargo xtask check-no-anyhow` blocks merge if any library crate pulls `anyhow`.
- `cargo xtask check-boundaries` blocks merge on forbidden inter-crate imports (see [05-module-responsibilities.md §10](05-module-responsibilities.md)).
- `cargo udeps` runs weekly and nags on unused dependencies.

---

## 11. What we do not test

- **Exact bit equality** with C++ xcfun outputs. Infeasible across compilers and libm variants.
- **`f32`-precision paths**. There are none; the crate is f64-only.
- **GPU kernels on a GPU without f64**. We refuse to run at batch-open time (see [06-cubecl-strategy.md §2](06-cubecl-strategy.md)).
- **ABI compatibility with non-Cargo build systems**. We publish a working `xcfun.h` and a `.so`/`.a`; how users integrate them is their concern.

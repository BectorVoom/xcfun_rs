use xcfun_core::{DensityVars, EvalMode, FunctionalId, VarType, XcError, taylorlen};

    order: Option<u32>,

Implement methods:

**new()**: Initialize empty state. `active = vec![]`, `depends = Dependency::empty()`, all Options are None, lengths are 0.

**set(&mut self, name: &str, weight: f64) -> Result<(), XcError>** (per D-18):
1. Check `aliases::lookup(name)` first
2. If alias found: for each `(id, w)` in alias.terms, call `self.add_functional(FunctionalImpl::from_id(*id), weight * w)`
3. If not alias: try `FunctionalId::from_name(name)`. If found, `self.add_functional(FunctionalImpl::from_id(id), weight)`
4. If neither: return `Err(XcError::UnknownName(name.to_string()))`
5. Private helper `add_functional(&mut self, f: FunctionalImpl, w: f64)`: push to `self.active`, update `self.depends |= f.depends()`

**eval_setup(&mut self, vars: VarType, mode: EvalMode, order: u32) -> Result<(), XcError>** (per D-13):
1. Validate: `if self.active.is_empty() { return Err(XcError::NotConfigured) }`
2. Validate: `if order > MAX_ORDER { return Err(XcError::InvalidOrder { order, mode, n_vars: vars.input_len() }) }`
3. Validate: `if !vars.provides().contains(self.depends) { return Err(XcError::InsufficientVars { vars, required: self.depends }) }`
4. Validate mode compatibility: Potential mode requires depends to NOT include LAPLACIAN or KINETIC (meta-GGA not supported for potential). If `mode == Potential && self.depends.intersects(Dependency::LAPLACIAN | Dependency::KINETIC) { return Err(XcError::UnsupportedMode { mode, depends: self.depends }) }`
5. Store `self.vars = Some(vars)`, `self.mode = Some(mode)`, `self.order = Some(order)`
6. Compute `self.input_len = vars.input_len()`
7. Compute `self.output_len` based on mode:
   - PartialDerivatives: `taylorlen(vars.input_len(), order as usize)` -- NOTE: the C++ uses the number of INPUT variables as the differentiation dimension
   - Potential: for LDA = 1 (energy) + number of spin channels (1 for closed-shell, 2 for A_B); for GGA = same + gradient contributions. For now: if A or N vars => 2 outputs (E, v_xc); if A_B or N_S vars => 3 outputs (E, v_alpha, v_beta).
   - Contracted: `1 << vars.input_len()` (2^n_vars Taylor coefficients)

**PartialDerivatives mode** (per D-14, EVAL-02):

The key challenge is the variable-pair loop with CTaylor const generic N. Since Rust requires const generics at compile time, use a match dispatcher on (n_vars, order) to call monomorphized inner functions.

For LDA with A_B (2 input variables), the approach is:
- n_vars = 2 (input_len for A_B = 2)
- N = 1 << n_vars = 4 (CTaylor array size)
- Output size = taylorlen(2, order)

Create a private helper that is generic over const N:
```rust
fn eval_partial_inner<const N: usize>(
    &self, input: &[f64], output: &mut [f64],
    vars: VarType, order: u32,
) -> Result<(), XcError>
```

Inside this helper:
1. Zero the output array
2. For each variable pair (i, j) where i <= j and the bit-index `idx = (1<<i) | (1<<j)` (or `1<<i` when i==j):
   a. Create `CTaylor<f64, N>` for each input variable, seeding the coefficient at `idx` position = 1.0 for the variables being differentiated
   b. Actually, the simpler approach for LDA: create input as `CTaylor<f64, N>` with `c[0] = input[k]` and `c[1<<k] = 1.0` for variable k (seeding first derivative). This is what the C++ does.
   c. Call `DensityVars::from_input(&ctaylor_inputs, vars)` -- DensityVars::from_input accepts `&[T]` where T: Num, so `&[CTaylor<f64, N>]`
   d. For each `(functional, weight)` in self.active: `result += T::from_f64(weight) * functional.energy(&dvars)`
   e. Extract all coefficients from the result CTaylor into the output array

CRITICAL implementation detail: The variable-pair loop from the design doc iterates over pairs (i,j) to compute mixed partial derivatives. But for a simpler initial approach that matches the C++ behavior:
- Create ALL input CTaylor variables at once, each seeded with `c[1<<k] = 1.0` for its index k
- Call `DensityVars::from_input` with the full set of seeded CTaylor inputs
- Evaluate once (not per-pair)
- The resulting CTaylor contains ALL derivatives up to the given order automatically

The output mapping is: `output[idx]` = `result.c[idx]` for all valid idx combinations up to the specified order. Use the taylorlen convention to map.

Implement the order/n_vars dispatcher:
```rust
match (n_vars, order) {
    (1, 0) => self.eval_partial_inner::<1>(input, output, vars, 0),
    (1, 1) => self.eval_partial_inner::<2>(input, output, vars, 1),
    (1, 2) => self.eval_partial_inner::<2>(input, output, vars, 2),
    (2, 0) => self.eval_partial_inner::<1>(input, output, vars, 0),
    (2, 1) => self.eval_partial_inner::<4>(input, output, vars, 1),
    (2, 2) => self.eval_partial_inner::<4>(input, output, vars, 2),
    // ... extend as needed for higher n_vars and orders
    _ => todo!("order {order} with {n_vars} vars not yet supported"),
}
```

Note: For 2 variables (A_B), N=4 (1<<2). The CTaylor<f64, 4> has indices: 0=constant, 1=var0(a), 2=var1(b), 3=var0*var1(mixed). The output mapping for order 2 is: [c[0], c[1], c[2], c[1|1]=c[1] second deriv? -- Actually, for order 2 with 2 vars, we need the second derivatives d2E/da2 and d2E/db2. But CTaylor<f64,4> with 2 bits only stores up to first-order mixed. For second order, we need CTaylor<f64, N> where N accommodates second-order terms.

RE-READ the CTaylor docs carefully. The C++ xcfun uses a specific indexing scheme. In Phase 1's implementation, `N = 1 << n_vars` gives FIRST-order capacity for n_vars variables. For higher orders, the N must be larger. Check how CTaylor is actually used in the C++ code base and in the Phase 1 implementation.

ACTUALLY: Read `crates/xcfun-ad/src/ctaylor.rs` to understand the exact indexing. The key insight from the Phase 1 decisions is: "N parameter represents array SIZE (power of 2) not variable count". For order `ord` with `nvar` variables, the C++ uses `taylorlen(nvar, ord)` elements. But CTaylor uses power-of-2 sizes. The mapping may be: for order 2 with 2 vars, we need N = taylorlen(2, 2) = 6, but N must be power of 2, so N = 8 (1 << 3).

Read ctaylor.rs carefully before implementing. The correct N for the const generic and the correct output mapping depend on understanding the actual array layout. The test data has 6 output elements for order 2 with 2 vars (A_B), so the implementation must produce exactly those 6 values in the right positions.

**Potential mode** (per D-12, EVAL-03):
For LDA (Dependency::DENSITY only), potential is simple:
- Seed CTaylor<f64, 2> with unit derivative on density variable
- output[0] = energy (c[0])
- output[1] = dE/d(alpha) (c[1]) -- this IS v_alpha for LDA
- For spin-polarized (A_B): repeat for beta, output[2] = v_beta

**Contracted mode** (per EVAL-04):
Input contains pre-computed Taylor coefficients. Create CTaylor from input directly (copy coefficients), evaluate functionals, extract all coefficients to output.

**kernels/lda.rs** (per D-15a, D-15b, D-15c, D-15e):

Implement cubecl kernel functions for each LDA functional. These are SEPARATE from the Functional trait implementations -- they use f64 directly (not generic T: Num) for batch evaluation on GPU/CPU.

CRITICAL cubecl coding rules (from D-15c and cubecl_error_solution_guide):
- Use `f64::exp(x)` NOT `x.exp()` for transcendentals
- Use `if` statements NOT `if` expressions: `let mut v = 0.0f64; if cond { v = a; }`
- Use `#[comptime]` for compile-time constants
- Use `ABSOLUTE_POS` for thread indexing, `terminate!()` for bounds
- NO `usize` in device code -- use `u32`
- Import as `use cubecl_core::{self as cubecl, prelude::*};`

1. `slaterx_batch_kernel`: Takes `density_a: &Array<f64>`, `density_b: &Array<f64>`, `output: &mut Array<f64>`, `#[comptime] c_slater: f64`, `#[comptime] n_points: u32`. For each thread idx: `output[idx] = -c_slater * (a^(4/3) + b^(4/3))`. For `x^(4/3)`, use `x * f64::cbrt(x)` (since cubecl may not support f64::powf -- per pitfall 5 in RESEARCH.md). If `f64::cbrt` is also unavailable, use `f64::exp(4.0/3.0 * f64::log(x))`.

2. `vwn5c_batch_kernel`: Takes density_a, density_b, output, plus VWN5 parameters as `#[comptime]` values. Computes n, zeta, r_s, sqrt(r_s), then calls inline VWN parameterized formula. Since `#[cube]` functions can call other `#[cube]` functions, factor out a `vwn_f_inline` helper. Output = n * eps.

3. `pw92c_batch_kernel`: Similar structure. Uses eopt parameterization and omega spin interpolation. All parameter constants passed as `#[comptime]`.

4. `pz81c_batch_kernel`: Uses branching on r_s. Use `if` statement pattern: `let mut eps = 0.0f64; if r_s > 1.0 { eps = low_density_formula; } else { eps = high_density_formula; }` -- note this is the if-STATEMENT pattern required by cubecl (not if-expression).

IMPORTANT: If the cubecl compile fails due to unsupported operations (like f64::cbrt, f64::powf, f64::log, f64::atan), work around:
- `cbrt(x)` = `f64::exp(f64::log(x) / 3.0)` if cbrt unavailable
- `x^(4/3)` = `f64::exp(4.0/3.0 * f64::log(x))`
- `atan(x)` -- if unavailable, this is a problem for VWN. May need to use a polynomial approximation or rethink the VWN cubecl kernel approach.

If VWN/PW92/PZ81 kernels prove too complex for cubecl (too many transcendentals), implement at minimum the SlaterX kernel (which only needs cbrt or exp+log) and leave the others as stubs with `todo!()`. Document in SUMMARY what worked and what didn't.

**runtime.rs** (per D-15f):

Implement buffer management and batch evaluation function using CpuRuntime.

```rust
use cubecl_core::prelude::*;
use cubecl_cpu::{CpuDevice, CpuRuntime};

/// Evaluate SlaterX for a batch of density points using cubecl CpuRuntime.
/// density_a and density_b are spin-alpha and spin-beta density arrays.
/// Returns energy density at each point.
pub fn evaluate_batch_slaterx(
    density_a: &[f64],
    density_b: &[f64],
) -> Vec<f64>
```

Implementation pattern (per D-15f, following cubecl_3d_dft.md):
1. `let device = CpuDevice::default();`
2. `let client = CpuRuntime::client(&device);`
3. Upload input buffers: `client.create(f64::as_bytes(density_a))`
4. Allocate output: `client.empty(n * core::mem::size_of::<f64>())`
5. Configure launch: `CubeDim::new_1d(256)`, `CubeCount::new_1d((n as u32 + 255) / 256)`
6. Launch kernel via `unsafe { slaterx_batch_kernel::launch_unchecked::<CpuRuntime>(...) }`
7. Read back: `let bytes = client.read_one(out_handle);`
8. Convert: `f64::from_bytes(&bytes).to_vec()`

Also implement `evaluate_batch_vwn5c`, `evaluate_batch_pw92c`, `evaluate_batch_pz81c` if the corresponding kernels compiled successfully. Otherwise, leave as stubs.

Add a higher-level dispatcher:
```rust
pub fn evaluate_batch_lda(
    functional_name: &str,
    density_a: &[f64],
    density_b: &[f64],
) -> Result<Vec<f64>, String>
```
That matches on functional_name to call the appropriate kernel wrapper.

### LDA path (Pass 1)

`launch_potential_lda` mirrors XCFunctional.cpp:637-670:
- Inlen + npot/inpos derivation per `inlen ∈ {1, 10, 2, 20}`.
- For each j ∈ 0..npot: pre-pack flat ct_in (inlen × 2 coefficients), seed VAR0=1 on `j*inpos`, launch `potential_lda_kernel` for every (id, weight) in `self.weights`, accumulate weighted `out[CNST]` (energy) + weighted `out[VAR0]` (potential) into `out[j+1]`.
- Energy at `out[0]` (last j's energy — same value for all j in LDA loop).

### GGA path (Pass 2)

`launch_potential_gga` mirrors XCFunctional.cpp:671-791:
- Per XCFunctional.cpp:671 structural invariant (no `else` between LDA and GGA blocks): `out[j+1]` is already populated with the LDA-direct ∂E/∂ρ term by `launch_potential_lda` BEFORE this fn runs.
- For each spin channel j ∈ 0..nspin and direction dir ∈ {x, y, z}:
  - Zero ct_in entirely, then re-pack with HESS_SLOT[src-1][dir] indirection (table inline at functional.rs).
  - Seed VAR1 = 1 on `(1 + dir) + active_offset` (active_offset = 10·j for spin-resolved).
  - Launch `potential_gga_kernel`, accumulate `Σ_dir Σ_id w_id · out[VAR0|VAR1]` into `divergence_accum`.
- Subtract `out[j+1] -= divergence_accum` IN PLACE, matching XCFunctional.cpp:720 / :785-787.

## Decisions Made

- **B5 path (a) — pragmatic interpretation:** Rather than spawn a brand-new C++ driver from xtask (which would require duplicating the entire xcfun-master cc::Build tree), the gen-potential-fixtures binary depends on validation/ as an optional dep. validation/ already cc-compiles all of xcfun-master and exposes CppXcfun. Functionally, the JSON IS still produced by an offline run of the cc-compiled C++ reference — fully aligned with the path (a) intent. Saves duplicating ~50 cc::Build::file lines.

- **Vars discriminant initially wrong:** First version of gen-potential-fixtures hard-coded `VARS_A_B_2ND_TAYLOR = 27`. Test run reported `inlen=10 outlen=2` (which is XC_A_2ND_TAYLOR, single-spin). Fix: corrected to 28 (per enums.rs:74 + xcfun.h:118 counting from XC_VARS_UNSET=-1 ⇒ XC_A=0 ⇒ ... ⇒ XC_A_B_2ND_TAYLOR=28). Caught immediately by length-mismatch assertion.

- **Per-functional vars routing in run_potential:** LDAs use Vars::A_B; GGAs use Vars::A_B_2ND_TAYLOR. metaGGAs (LAPLACIAN/KINETIC deps) skipped per D-13 (Phase 4 scope). Routing applied via FUNCTIONAL_DESCRIPTORS.depends inspection.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Missing gnn/gns/gss derivation in build_xc_a_b_2nd_taylor**

- **Found during:** Task 3 — `potential_parity_100` initial run with first iteration of fixture generator.
- **Issue:** `build_xc_a_b_2nd_taylor` populated `gaa`, `gab`, `gbb`, `lapa`, `lapb` from input slots but did NOT derive `gnn`, `gns`, `gss` (the helper sibling `build_xc_a_b_gaa_gab_gbb` had this derivation but the 2ND_TAYLOR variant was missing it). LYPC reads `d.gnn` directly (lyp.rs:140, :195), so under Mode::Potential the LYPC potential drifted by ~1e-2 to ~1e-6 rel_err vs the C++ reference (60/100 records failing). PBEX/BECKEX/PW91X/B97X passed because they don't read `d.gnn`.
- **Fix:** Appended the 3-line gnn/gss/gns derivation block from `build_xc_a_b_gaa_gab_gbb` (densvars/build.rs:246-258) to `build_xc_a_b_2nd_taylor` after the cross-spin gab computation. Operation order: `gnn = gaa + 2·gab + gbb` (left-to-right, no mul_add per ACC-06), `gss = gaa - 2·gab + gbb` (reuses `t1 = 2·gab`), `gns = gaa - gbb`.
- **Files modified:** `crates/xcfun-eval/src/density_vars/build.rs`
- **Verification:** After fix, `potential_parity_100` passes 100/100 at strict 1e-12. Validation harness runs LYPC at 30k records: 0 failures. No regression in pre-existing tests (29 xcfun-eval tests still pass).
- **Committed in:** `dcf86a6` (Task 3 commit, intertwined with the Mode::Potential CLI + B5 generator)

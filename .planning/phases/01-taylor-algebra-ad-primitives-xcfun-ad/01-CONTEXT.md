# Phase 1: Taylor Algebra & AD Primitives (`xcfun-ad`) - Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Build the `xcfun-ad` crate: `CTaylor<T, N>` bit-flag-indexed multilinear polynomial (order 0..=7 storage, supporting derivative orders 0..=6 plus one contracted-mode bit), `Num` trait for `f64` + `CTaylor<f64, N>`, and every `*_expand` scalar series function from `xcfun-master/external/upstream/taylor/tmath.hpp` ported byte-for-byte. Composed elementary functions (`reciprocal`, `sqrt`, `exp`, `log`, `pow`, `powi`, `erf`, `asinh`, `atan`) built on top via series composition. Property tests, golden `f64::to_bits` tests against the C++ reference, and a criterion baseline.

Out of scope for this phase: functional ports, `DensVars`, registry, dispatch, kernels, GPU — all downstream.

</domain>

<decisions>
## Implementation Decisions

### Storage & const-generic bounds
- **D-01:** `CTaylor<T, const N: usize>` stores coefficients in a single `pub c: [T; 1 << N]`, `#[repr(C)]`. Stack only, no heap, no `Box`, no `Vec`. Valid `N` range is `0..=7`.
- **D-02:** `N ≤ 7` enforced via a sealed `ValidN<N>` marker trait implemented only for `N` in `0..=7` — stable-Rust compatible, no `generic_const_exprs` nightly feature required. Chosen over `Assert<{N<=7}>: True` (nightly) and runtime asserts (too lax).
  - *[auto-selected — recommended]*
- **D-03:** `Copy + Clone + Debug + PartialEq` derived. No `Hash`, no `Eq` (f64 is not Eq).

### `Num` trait & numeric backbone
- **D-04:** Custom `Num` trait with `Add`, `Sub`, `Mul`, `Div`, `Neg`, `reciprocal`, `sqrt`, `exp`, `log`, `pow`, `powi`, `erf`, `asinh`, `atan` plus `ZERO` and `ONE` constants. Implemented for `f64` and for `CTaylor<f64, N>` (via blanket-impl gated on `ValidN<N>`).
- **D-05:** `Num` is NOT `num-traits::Float`. Rationale locked by design D2 in `docs/design/12-design-decisions.md`. Avoids bringing in `is_nan`, `classify`, `floor`, etc., which are meaningless on polynomials.
- **D-06:** `f64` is the only scalar implementation. `f32` intentionally unimplemented; design D4.

### `CTaylor::mul` recursion structure
- **D-07:** `CTaylor::mul` ports `ctaylor_rec<T, Nvar>::multo` from `ctaylor.hpp` line-for-line, preserving the `P_N = P_{N-1} + x_N · R_{N-1}` recursion on the highest bit. The Rust form uses `#[inline]` helper functions parameterised by a const `N`, one per order (0..=7), mechanically derived from the C++ recursion. No re-association, no FMA, no parallel accumulation.
- **D-08:** Operation order in every non-trivial expression preserved via explicit `let` bindings matching the C++ intermediate variables. Multi-term sums never collapsed into a single `a + b + c + d` expression — always bound step-by-step so the compiler's traversal is deterministic.

### `*_expand` scalar series ports
- **D-09:** One Rust module per C++ function: `inv_expand`, `exp_expand`, `log_expand`, `pow_expand`, `sqrt_expand`, `cbrt_expand`, `erf_expand`, `gauss_expand`. Every `*_expand` writes into a caller-provided `&mut [f64; 8]` (size = `N_MAX + 1`, stack-only) — no heap allocation inside expansions.
- **D-10:** Each `*_expand` body is a textual port of the C++ recurrence with a comment block citing the upstream line range (e.g., `// tmath.hpp:124-145: pow_expand`). This is the primary code-review anchor for P9 (SUMMARY.md pitfall — `*_expand` coefficient layout miscopied).
- **D-11:** Preconditions on `*_expand` inputs (e.g., `x0 > 0` for `log_expand`) are enforced with `assert!` (NOT `debug_assert!`), keeping the check active in release builds. Catches the silent-NaN failure mode P10 described in SUMMARY.md.
  - *[auto-selected — recommended per P10]*

### Composed elementary functions (on `CTaylor<f64, N>`)
- **D-12:** `reciprocal`, `sqrt`, `exp`, `log`, `pow`, `powi`, `erf`, `asinh`, `atan` implemented as `CTaylor` → scalar `*_expand` → `ctaylor_rec::compose`-equivalent series composition. Matches `ctaylor_math.hpp` operation order verbatim.
- **D-13:** `pow(x, integer)` prefers `powi` over `pow` whenever the exponent is an integer literal at the call site. Standard lint in Phase 2+ enforces this for functional bodies; Phase 1 documents the convention in the crate root.

### `no_std` and libm dependency
- **D-14:** `xcfun-ad` has a default feature `std` (default-on) and an optional feature `libm`. With `std` on, `f64` math uses `std::f64::{sqrt, exp, log, powf, ...}`. With `std` off and `libm` on, `f64` math routes through the `libm` crate. With neither, `Num for f64` is disabled (kernel crates will compile against cubecl intrinsics directly at Phase 7, not via `xcfun-ad`).
  - *[auto-selected — recommended; keeps no_std story open without blocking Phase 1]*

### Golden fixture generation (C++ parity gate)
- **D-15:** Pre-generated fixtures committed to `xcfun-ad/tests/fixtures/*.bincode`, regenerated by `cargo xtask regen-ad-fixtures` which compiles a small C++ driver linking `xcfun-master/external/upstream/taylor/` and emits deterministic records. CI does NOT regenerate — it consumes the committed fixtures. Preserves hermetic `cargo build`; only Phase 0 maintainers need a C++ toolchain.
  - *[auto-selected — recommended; alternative (build.rs invoking cc + running) breaks hermeticity]*
- **D-16:** Fixture format: `bincode::serialize` of `Vec<FixtureRecord>` where `FixtureRecord` is `{ op: String, n_var: u8, inputs: Vec<f64>, coeffs: Vec<f64> }`. Parallel JSON manifest (`fixtures.json`) lists every record for debuggability and version pin (`xcfun_version_git_sha`).
- **D-17:** Fixture size budget: ≤ 1 MB total. If the total exceeds this, drop redundant inputs (not coverage breadth).

### Property tests & bench baseline
- **D-18:** Property tests (`proptest` 1.11) with ≥ 10 000 iterations per property: commutativity, associativity, distributivity, ring axioms, `exp(x) · exp(-x) ≈ 1` (rel. err. ≤ 1e-13 with small random inputs), `log(exp(x)) ≈ x`, `sqrt(x)² = x`, `pow(x, a) · pow(x, -a) ≈ 1`, Leibniz product rule on `VAR0` coefficient.
- **D-19:** Criterion baseline: `CTaylor::<f64, N>::mul_assign` for `N ∈ {2, 3, 4, 5, 6}` and composed `exp`/`log`/`pow` at `N = 4`. Baseline recorded at Phase 1 completion; no regression gate in v1 (PERF-01 deferred to v2 per REQUIREMENTS.md).

### Floating-point hygiene (phase-1-local guardrails)
- **D-20:** Crate-root `#![deny(clippy::float_arithmetic_side_effects)]` is NOT set (would block legitimate ops). Instead, functional-body `mul_add` lint is introduced in Phase 2. Phase 1 keeps a single assertion in CI: the release build of `xcfun-ad` must link without `fma` intrinsic calls in `CTaylor::mul`'s object file (checked by a `cargo asm` spot-check in CI, grep for `vfmadd` / `fmadd`).
- **D-21:** `-Cllvm-args=-fp-contract=off` set in `.cargo/config.toml` `[build]` for the release profile (Phase 0 carryover — applies to this crate).

### Testability seams
- **D-22:** `pub mod for_tests` gated behind `feature = "testing"` exposes `raw_coeffs<T, const N: usize>(&CTaylor<T, N>) -> &[T; 1 << N]` and direct-construction helpers. Not exposed in the default build surface. Downstream crates (`xcfun-core`) depend on `xcfun-ad` with `features = ["testing"]` only in `[dev-dependencies]`.

## Existing Code Insights

### Reusable Assets
- **`xcfun-master/external/upstream/taylor/`** — C++ source the Rust port mirrors verbatim. Read, port, diff.
- **Phase 0 scaffolding** (not yet landed) — workspace, `rust-toolchain.toml`, `deny.toml`, `.cargo/config.toml` with `-Cllvm-args=-fp-contract=off`. Phase 1 assumes these are in place.
- **`xtask` skeleton** (Phase 0) — provides the harness for `regen-ad-fixtures` subcommand added here.

### Established Patterns
- **One-source rule** (design D1): CTaylor is a textual port; Rust idioms (fluent APIs, iterator chains) are secondary to algorithmic identity.
- **Stack-only storage** (design principle): every new type must satisfy `Copy` when the underlying `T` is `Copy`; fixed-size arrays only.
- **No `unsafe`** inside `xcfun-ad`: crate root `#![forbid(unsafe_code)]`. If an unsafe optimization is later needed, it must be contained in a justified module with a documented invariant.

### Integration Points
- `xcfun-core` (Phase 2) consumes `CTaylor<f64, N>` via `pub use xcfun_ad::*` in `xcfun-core::ad`. This is the only downstream consumer at Phase 2.
- `xcfun-kernels` (Phase 6) will re-implement a device-side `CTaylorDev<F, N>` using the same bit-flag indexing. Phase 1 must therefore keep the bit-index API (`CNST = 0`, `VAR0 = 1 << 0`, …, `VAR6 = 1 << 6`) as `pub const` items so the kernel crate can mirror them without duplication.

</code_context>

<specifics>
## Specific Ideas

- Match the C++ author's naming: `ctaylor_rec::multo` → `mul_into` on the Rust side (verb mirror; `into` signals the destination-first convention). Keep `VAR0..VAR6`, `CNST` identifiers byte-identical to ease cross-reference.
- Coefficient extraction: prefer the literal `t.c[CNST | VAR0 | VAR1]` pattern over a fluent builder. Bit-OR indices match the C++ read sites line-for-line.
- Every series-expansion module carries a header comment with three items: (1) the upstream tmath.hpp line range, (2) the mathematical identity in LaTeX notation, (3) the preconditions (`x0 > 0`, etc.). This is the reviewer's checklist.

</specifics>

<deferred>
## Deferred Ideas

- **SIMD vectorisation of `CTaylor::mul`** — deferred to potential v2 work once the 1e-12 baseline is locked. Premature now (risk of reassociation).
- **`CTaylor<CTaylor<f64, M>, N>` nested types** — design considered but not implemented; no downstream consumer needs order-beyond-6 and nesting introduces new ordering questions.
- **Property-test reduction of rel-error budget below 1e-13** — only achievable at ULP level; not a Phase 1 goal.
- **Crate-level benchmark regression gate** — deferred to v2 (PERF-01).
- **`no_std` + no `libm` pure-polynomial mode** — would require `Num` only for `f64` with scalar ops delegated to caller-supplied math, which breaks the ergonomic story. Feasible but not needed.

# Requirements: xcfun_rs

**Defined:** 2026-04-19
**Core Value:** Every functional must produce numerical output matching C++ xcfun within relative error <= 1.0 x 10^-12, across all evaluation modes and derivative orders.

## v1 Requirements

### AD Engine

> **Cubecl pivot (2026-04-19 PM):** Phase 1 was rewritten from a hand-Rust `CTaylor` port to a cubecl-native `#[cube]` type. AD-01..AD-06 are restated below for the cubecl implementation. See `.planning/phases/01-taylor-algebra-ad-primitives-xcfun-ad/01-CONTEXT.md` for the locked decisions. AD-01 is un-ticked because the prior `[x]` (commit `c7a3f46`) is being reverted as part of the new plan.

- [x] **AD-01**: `CTaylor<F: Float, const N: u32>` is a pure `#[cube]` type for `N ∈ 0..=7`, backed by a cubecl `Array<F>` of length `1 << N`, bit-flag-indexed (`CNST = 0`, `VAR0 = 1`, …, `VAR6 = 64` exposed as host `pub const`). Algorithmic identity with `xcfun-master/external/upstream/taylor/ctaylor.hpp` is preserved inside the `#[cube]` source.
- [x] **AD-02**: `xcfun-ad` exposes `#[cube] fn` operations — `ctaylor_add`, `ctaylor_sub`, `ctaylor_mul`, `ctaylor_div`, `ctaylor_neg`, `ctaylor_reciprocal`, `ctaylor_sqrt`, `ctaylor_exp`, `ctaylor_log`, `ctaylor_pow`, `ctaylor_powi`, `ctaylor_erf`, `ctaylor_asinh`, `ctaylor_atan` — generic over `F: Float` (cubecl's `Float` trait, not the retired in-house `Num`). Constants `ZERO` and `ONE` available via `F::new(0.0)` / `F::new(1.0)` per cubecl idiom. (Plan 01-06: 9 composed fns shipped in `crates/xcfun-ad/src/math.rs`; `ctaylor_div` deferrable — expressible as `ctaylor_pow(x, F::new(-1.0))`.)
- [x] **AD-03**: `#[cube] fn ctaylor_mul` accumulates coefficients in exactly the recursion order of `ctaylor_rec<T,Nvar>::multo` (verbatim port, not cubecl-idiomatic rewrite). CI evidence (asm spot-check or equivalent) confirms cubecl-cpu's MLIR JIT does not reorder or fuse operations on the f64 path; if reordering is unavoidable, plan-phase escalates rather than widening the 1e-12 tolerance.
- [x] **AD-04**: every `*_expand` function (`inv_expand`, `exp_expand`, `log_expand`, `pow_expand`, `sqrt_expand`, `cbrt_expand`, `erf_expand`, `gauss_expand`) from `xcfun-master/external/upstream/taylor/tmath.hpp` has a `#[cube] fn <name>_expand<F: Float>` port writing into a length-8 `Array<F>`. Each port carries a doc-comment header citing upstream line range, mathematical identity, and preconditions enforced via cubecl `assert!`-equivalents active in release builds.
- [x] **AD-05**: `CTaylor` algebra and `*_expand` ports pass golden tests against the C++ reference at relative error ≤ **1e-12** (matching the project Core Value, executed via `cubecl-cpu`) for orders 0..=3 on a fixed-seed set of inputs. Fixtures regenerated by `cargo xtask regen-ad-fixtures` from a C++ driver linking `xcfun-master/external/upstream/taylor/`. (Plan 01-05: golden_mul.rs; Plan 01-06: golden_expand.rs + golden_composed.rs — 598 records covering mul / expand / composed; cbrt/erf/gauss relaxed to 1e-7 per upstream polyfill disclosures.)
- [x] **AD-06**: AD engine property tests (ring axioms, exp/log round-trip, sqrt-squared invariance, Leibniz product rule, ≥ 11 properties) run ≥ 10 000 iterations per property using the **batch-per-property kernel pattern** (proptest generates 10k inputs upfront, single cubecl-cpu kernel evaluates all 10k, results aggregated host-side) without failure. (Plan 01-07: `crates/xcfun-ad/tests/proptest_algebra.rs` — 11 tests × 10 000 iters = 110 000 total, zero failures on cubecl-cpu.)

## Out of Scope

Explicitly excluded; documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Libxc-scale functional coverage (~400 functionals) | Scope explosion; xcfun_rs competes on derivative order (up to 6), not coverage — users pick Libxc for breadth |
| Bit-identical output with C++ xcfun | Infeasible across libm / CUDA math / WGSL variants; 1e-12 relative tolerance is the stated contract |
| `f32` numerical path | Cannot meet 1e-12 tolerance (f32 unit roundoff ≈ 6e-8) |
| Extended precision (`f128`, `long double`) | Reference runs on `double`; no host consumer need |
| Differentiation order > 6 | Reference fixes `XCFUN_MAX_ORDER = 6`; raising would require re-verifying the coefficient algebra |
| Third-party AD frameworks (Enzyme, hyperdual, `ad` crate) | None replicate xcfun's bit-flag multilinear polynomial algebra; parity unprovable |
| User-supplied Rust functionals with AD | Only the library's 78 built-in functionals are differentiated; plugin API would break AD-engine-as-private-surface contract |
| VV10 non-local correlation | Separate library concern (gpu4pyscf / Libxc handle it) |
| D3/D4 dispersion correction | Out of XC-functional scope; use `dftd4` in parallel |
| ML-based functionals (Skala, libNXC) | Different dependency graph; users link alongside if needed |
| Libxc-compatibility shim | Different product; forces naming/normalisation conflicts |
| Distributed / multi-node evaluation | Density grids are local; host MPI layer owns distribution |
| Stream-overlapped async GPU in v1 | < 20 % throughput gain for 3× API complexity; revisit in v2 |
| Symbolic / rational output | No host consumer need |
| Pre-API-v2 C headers | Target current xcfun ABI only |
| `rayon` in library crates | Caller owns grid parallelism; library parallelism introduces non-determinism in active-functional accumulation order |
| Runtime-loadable custom functionals | Breaks hermetic `cargo build` + precompiled kernel dispatch table |

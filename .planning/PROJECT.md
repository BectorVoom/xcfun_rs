# xcfun_rs

## What This Is

A Rust-from-scratch reimplementation of the xcfun exchange–correlation functional library for density functional theory (DFT). It reproduces the full public C API of the C++ reference (`xcfun-master/`) — 78 functionals, 50+ aliases, 31 variable combinations, 3 evaluation modes, arbitrary derivative orders 0–6 — and extends it with a native Rust API, a C ABI drop-in replacement, Python bindings (pyo3), and a unified CPU/GPU batch evaluation engine implemented in `cubecl` so a single kernel source covers `CpuRuntime`, `CudaRuntime`, and `WgpuRuntime`. The intended audience: computational-chemistry code authors who need a memory-safe, GPU-capable XC library without giving up bit-level parity with the C++ reference.

## Core Value

Every functional must produce numerical output matching C++ xcfun within relative error ≤ 1.0 × 10⁻¹², across all evaluation modes and derivative orders.

### Out of Scope

<!-- Explicit boundaries. Includes reasoning to prevent re-adding. -->

- Bit-identical output with C++ xcfun — infeasible across libm/CUDA math variants; 1e-12 relative tolerance is the stated contract
- Distributed / multi-node evaluation — density grids are local by construction; callers handle distribution
- Stream-overlapped async GPU path in v1 — < 20 % throughput gain for 3× API complexity
- `f32`-precision numerical path — cannot meet 1e-12 tolerance (f32 unit roundoff ≈ 6e-8)
- `f128` / `long double` extended precision — reference runs on `double`; no stated need
- Third-party AD frameworks (Enzyme, hyperdual, `ad` crate) — none replicate xcfun's bit-flag multilinear polynomial algebra; parity would be unprovable
- Differentiation order > 6 — reference fixes `XCFUN_MAX_ORDER = 6`; raising would require re-verifying coefficient algebra
- User-supplied Rust functionals with AD — only library's 78 built-in functionals are differentiated
- Symbolic output — `f64` numerical only
- Pre-API-v2 C headers — target exactly the current xcfun ABI

## Constraints

- **Accuracy**: Output must match C++ xcfun within relative error ≤ 1.0 × 10⁻¹² on every `(functional, vars, mode, order, density point)` tuple — the primary contract, non-negotiable
- **Compatibility**: C FFI must be a drop-in replacement for `xcfun-master/api/xcfun.h` — every declared symbol present with matching signature
- **Rust Edition**: 2024, MSRV 1.85 — required for current const-generic features
- **Compiler flags**: no `-Cfast-math`, no reassociation flags; `RUSTFLAGS` empty in CI — fast-math would break the accuracy contract
- **Tech stack**: `thiserror` 2.0.18 (library errors), `anyhow` 1.0 (app boundaries only — no library depends on it), `bitflags` 2.10.0, `cubecl` pinned at `=0.10.0-pre.3`, `pyo3` 0.28.3 + `numpy` 0.28.0, `cbindgen` 0.29.2, `criterion` 0.8.2, `approx` 0.5 — pinned in CLAUDE.md
- **GPU**: `cubecl` for CPU / CUDA / Wgpu backends; f32 never on the numerical path; Wgpu validated at relaxed 1e-9 tolerance (device `erf` variance), CUDA and CPU at 1e-12
- **Memory**: zero heap allocation on the per-point hot path; batch lifecycle allocates device buffers with powers-of-two growth
- **Licensing**: MPL-2.0 (inherited from xcfun reference)

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|

| Single `cubecl` kernel source for CPU and GPU (not two implementations) | One source ⇒ one algorithm ⇒ simpler parity verification; avoids drift | — Pending |

| `thiserror` in all library crates, `anyhow` only at application boundaries | Typed pattern-matchable errors for libraries, ergonomic propagation for apps | — Pending |
| `cubecl` pinned at `=0.10.0-pre.3` | Pre-release API; any bump re-runs the full validation harness | — Pending |
| Functional registry generated by `xtask codegen`, checked into git | Keeps `cargo build` hermetic — no C++ toolchain required to build | — Pending |

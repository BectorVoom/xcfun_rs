<objective>
Parallelize the tier-2 validation harness (`validation/` crate) so the order-3 full sweep
(~76 functionals × 4 orders × ~10 000 grid points) completes in roughly `1 / N_cores` of
the current wall-clock time. The user invocation that motivates this work:

```
cargo run -p validation --release -- --backend cpu --order 3 --resume --filter '.*' \
  2>&1 | tee /tmp/04-10-signoff-sweep.log
```

The numerical contract (1e-12 vs C++ xcfun) is non-negotiable; parallelism must not change
any record's value. Per CLAUDE.md hard rules: NO `rayon`, NO `crossbeam`, NO `unsafe impl Send`
for the FFI handle, `std`-only orchestration. Each worker constructs its own `CppXcfun`
inside the worker closure (the FFI handle is `!Send + !Sync` per validation/src/ffi.rs:28-30).

Purpose: unblock the user's order-3 sign-off sweep (currently serial) without touching the
numerical kernels.

Output: a `--jobs N` CLI flag, a `std::thread::scope` orchestrator inside each of the three
driver entry points (`run`, `run_potential`, `run_contracted`), a single-writer drain loop
that preserves the existing `Report.push_with_sink` matrix + JSONL flush semantics, and a
parity test that proves `--jobs 1` and `--jobs 4` produce identical output.
</objective>

1. **JSONL record order does NOT matter to any consumer.**
   - `grep -rn "report.jsonl" --include="*.rs" --include="*.sh" --include="*.toml" --include="*.yml"`
     across `xtask/`, `.github/`, `validation/`, `crates/` shows the only programmatic
     readers are `validation::report::read_completed_tuples` (builds a `HashSet<TupleKey>`
     — order-agnostic) and `validation::report::rebuild_matrix_from_jsonl` (builds a
     `HashMap<(String, u32), CellSummary>` aggregation — order-agnostic).
   - `report.html` is rendered from `Report.matrix` (a `BTreeMap` keyed by
     `(functional, order)` — alphabetical).
   - The `xtask/src/bin/regen_registry.rs::parse_jsonl` is a DIFFERENT JSONL (the C++
     extractor stream `xcfun-master/...`), not `validation/report.jsonl`.
   - The "byte-for-byte JSONL invariant expected by the merge gate" comment in
     `validation/src/main.rs:77-78` is aspirational; **no merge-gate script reads
     `report.jsonl` line-by-line in order.** No `.github/` workflows file exists
     in this repo (verified by `ls /home/chemtech/workspace/xcfun_rs/.github`
     showing only `assets`, `Cargo.toml`, `src` from `xtask`-adjacent layout — no
     `workflows/` directory and no shell scripts under `scripts/`).
   - **Verdict:** parallel emission with interleaved records is safe. `--jobs 1` MUST
     still produce sequential records (so a `--serial` flag is unnecessary; users who
     want byte-stable output for off-line diff tooling pass `--jobs 1`).

2. **xcfun-master C++ FFI has ONE mutable global on the eval path, and it is safe.**
   - `grep -n "static\b\|thread_local\b"` over
     `xcfun-master/src/{XCFunctional.cpp,XCFunctional.hpp,xcint.cpp,xcint.hpp,functional.hpp}`:
     the only non-template, non-const mutable static reachable from `xcfun_new`/
     `xcfun_set`/`xcfun_eval_setup`/`xcfun_eval` is **`static bool is_setup = false;`
     at xcint.cpp:138** inside `xcint_assure_setup()`.
   - `is_setup` is a one-shot init guard. Once set, it is read-only forever. The
     guarded body (`retarded_helper<0, XC_NR_FUNCTIONALS>::doit()` +
     `xcint_parameter_setup_helper`) initialises read-only template-static
     descriptor tables.
   - The race window: two threads see `is_setup == false` simultaneously on the very
     first call ever. Both run the body. The body is **idempotent** — it writes the
     same descriptor data twice — but the descriptor writes are not atomic, so a
     concurrent reader could see a torn pointer.
   - **Mitigation (in this plan):** `main.rs` constructs one `CppXcfun` on the main
     thread BEFORE spawning workers. The `xcfun_new()` constructor calls
     `xcint_assure_setup()` (XCFunctional.cpp:359). After that single call, every
     subsequent thread observes `is_setup == true` and skips the body. The pre-warm
     handle is dropped immediately; it leaves no thread-affinity.
   - All other reachable state is per-handle (`XCFunctional *` is a heap-owned object
     constructed inside `xcfun_new()` — no global mutable state).
   - **Verdict:** safe to parallelise after a single pre-warm `CppXcfun::new(); drop(it);`
     on the main thread. No global mutex on the FFI calls is needed.
</verified_findings>

    use std::sync::mpsc;
    use std::collections::VecDeque;

    let n_workers = cfg.jobs.get().min(jobs.len()).max(1);
    let queue: Arc<Mutex<VecDeque<Job>>> =
        Arc::new(Mutex::new(jobs.into_iter().collect()));
    let (tx, rx) = mpsc::channel::<ReportRecord>();

    let mut handles = Vec::with_capacity(n_workers);
    for _ in 0..n_workers {
        let q = Arc::clone(&queue);
        let tx = tx.clone();
        let grid = grid; // &[GridPoint], Copy of slice ref
        handles.push(s.spawn(move || -> Result<()> {
            // Each worker constructs its own CppXcfun handle (NOT Send).
            // Loop: pop a Job; run its full per-tuple body; emit each
            // ReportRecord via tx; pop next; on empty queue, return.
            loop {
                let job = match q.lock().unwrap().pop_front() {
                    Some(j) => j,
                    None => break,
                };
                let mut cpp = CppXcfun::new();
                // Per-tuple set-up: cpp.set + cpp.eval_setup + length checks
                // (factor a `run_one_tuple_*` helper per driver that returns
                // an iterator of ReportRecords or sends them on tx).
                run_one_tuple(&job, grid, &mut cpp, &tx)?;
            }
            Ok(())
        }));
    }
    drop(tx); // drop the orchestrator's clone so rx returns Err once all workers exit

    // Single-writer drain. This is the ONLY thread that touches `report` and
    // `cfg.sink`, preserving today's per-line-flush + matrix invariant.
    while let Ok(rec) = rx.recv() {
        report.push_with_sink(rec, cfg.sink.as_deref_mut())?;
    }

    for h in handles {
        h.join().map_err(|e| anyhow::anyhow!("worker panicked: {:?}", e))??;
    }

    //!
    //! Quick task 260430-4x7 — proves `--jobs 1` and `--jobs N>1` produce

    //! AND that the parallel JSONL on disk is content-equivalent to the
    //! serial JSONL after sort.

    use std::num::NonZeroUsize;
    use validation::driver::{run_with_mode_cfg, HarnessMode, ReportRecord, RunConfig};
    use validation::fixtures::generate_grid;
    use validation::report::{JsonlSink, TupleKey};

    fn run_to_jsonl(jobs: usize, filter: &str, path: &str) -> validation::driver::Report {
        let grid = generate_grid();
        let small_grid: Vec<_> = grid.into_iter().take(128).collect();
        let regex = regex::Regex::new(filter).unwrap();
        let mut sink = JsonlSink::create(path).unwrap();

            0,                                 // max_order

            HarnessMode::PartialDerivatives,
            &mut cfg,
        ).unwrap()
    }

    fn parse_jsonl(path: &str) -> Vec<ReportRecord> {
        use std::io::{BufRead, BufReader};
        let f = std::fs::File::open(path).unwrap();
        BufReader::new(f).lines().filter_map(|l| {
            let l = l.ok()?;
            if l.is_empty() { return None; }
            serde_json::from_str::<ReportRecord>(&l).ok()
        }).collect()
    }

    fn key(r: &ReportRecord) -> (String, String, u32, u32, usize, usize) {
        (r.functional.clone(), r.vars.clone(), r.mode, r.order, r.point_idx, r.element_idx)

        let parallel_path = dir.path().join("parallel.jsonl");

        let _ = run_to_jsonl(4, "xc_slaterx", parallel_path.to_str().unwrap());

        let mut s = parse_jsonl(serial_path.to_str().unwrap());
        let mut p = parse_jsonl(parallel_path.to_str().unwrap());
        assert_eq!(s.len(), p.len(), "JSONL record count mismatch");
        s.sort_by_key(|r| key(r));
        p.sort_by_key(|r| key(r));

            assert_eq!(serde_json::to_string(a).unwrap(),
                       serde_json::to_string(b).unwrap(),
                       "JSONL record content mismatch at key {:?}", key(a));
        }
    }

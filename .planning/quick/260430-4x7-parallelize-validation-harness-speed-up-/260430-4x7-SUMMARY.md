# Quick Task 260430-4x7: Parallelize validation harness

## Outcome

The motivating invocation now accepts a `--jobs N` flag (default `auto` →
`std::thread::available_parallelism()`) and dispatches each
`(functional, vars, mode, order)` tuple to a worker pool inside
`std::thread::scope`. Per-worker `CppXcfun` handles preserve the
`!Send + !Sync` FFI contract; an MPSC drain on the orchestrator preserves
the existing `Report.matrix` + per-line `JsonlSink::write_record` flush.

The user's command:

```
cargo run -p validation --release -- --backend cpu --order 3 --resume --filter '.*' \
  2>&1 | tee /tmp/04-10-signoff-sweep.log
```

Now picks up parallelism by default. `--jobs 1` reproduces the legacy
serial path byte-for-record-content for off-line diff tooling.

## Smoke wall-times (from commit `8c59675`)

Filter: `xc_slaterx + xc_pbex`, order 0, 20 000 records.

| `--jobs` | Wall |
|----------|------|
| 1        | 13.59 s |
| 4        | 8.19 s  |

1.66× on 4 jobs at order 0 — speedup grows with order because per-tuple
work scales as O(grid × outlen), and `outlen = taylorlen(inlen, order)`
is super-linear in `order`. The user's full order-3 sweep should see a
larger speedup since the worker-bound work dominates orchestration
overhead.

## Verified findings (frozen at plan time)

1. **JSONL record order does NOT matter to any consumer.**
   `read_completed_tuples` builds a `HashSet<TupleKey>` and
   `rebuild_matrix_from_jsonl` aggregates into a `HashMap<(String, u32),
   CellSummary>` — both order-agnostic. `report.html` renders from a
   `BTreeMap` keyed by `(functional, order)` (alphabetical). No CI / shell
   / xtask script reads `validation/report.jsonl` line-by-line in order.
   The "byte-for-byte JSONL invariant" comment in
   `validation/src/main.rs:77-78` was aspirational — no merge gate exists
   in the repo. Parallel emission with interleaved records is safe.

2. **xcfun-master C++ FFI has ONE mutable global on the eval path, and
   it is safe.** `static bool is_setup` at `xcint.cpp:138` is a one-shot
   init guard inside `xcint_assure_setup()`. Body is idempotent (writes
   the same descriptor data twice on a race) but not atomic, so a
   concurrent first-ever call could observe torn pointers. **Mitigation
   in main.rs:** construct one `CppXcfun` on the main thread before
   `std::thread::scope` spawns workers; this drives `is_setup → true`
   before any worker can race. After that, all C++ state is per-handle.
   No global mutex on FFI calls is needed.

## Architecture (final)

- `validation/src/main.rs` — `--jobs auto|N` flag, FFI pre-warm via a
  scoped `CppXcfun::new()` + `drop`.
- `validation/src/driver.rs` — `RunConfig.jobs: NonZeroUsize` field
  threaded through; per-driver `JobPD`/`JobPot`/`JobCon` types; per-driver
  `run_one_tuple_*` helpers take an `emit: FnMut(ReportRecord)` closure
  (no direct contact with `Report` / `JsonlSink`); generic
  `parallel_dispatch<J, W>` orchestrator. `cfg.jobs == 1` short-circuits
  to the serial path; `cfg.jobs > 1` spawns N workers in
  `std::thread::scope`, draining records via `mpsc::channel`.
- Excluded markers and resume-skip log entries are emitted from the
  orchestrator thread BEFORE any worker spawns; workers see only "real
  work" jobs.

## CLAUDE.md hard rules respected

- No `rayon`, no `crossbeam` — `std::thread::scope` + `std::sync::mpsc`
  + `Arc<Mutex<VecDeque<J>>>` only.
- No `unsafe impl Send` / `unsafe impl Sync` for `CppXcfun` — workers
  construct their own handles.
- `tempfile = "3"` lives in `[dev-dependencies]` only — does not enter
  the production validation binary. App-boundary scope preserved.
